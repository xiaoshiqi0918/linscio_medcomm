# MedComm 多模态深度实施计划（个人版）

> 定位：在现有「图像生成 / 文章级 visual continuity / 部分 ComfyUI 字段」之上，把 **条漫、分镜、系列图** 等多形式做到可依赖生产，而非单次试玩。  
> 关联设计：见同目录 `ComfyUI工作流编排设计方案.md`。

---

## 一、现状摘要（代码与产品）

- 后端 `imagegen` 已支持多提供商、文章级 `visual_continuity_prompt`、系列种子等；工作流 State 中已有 Comfy 相关字段（`comfy_workflow_path`、`comfy_base_url`、节点 id 等）。
- 前端有独立「图像生成」入口；编辑器内配图与多形式（comic / storyboard 等）已有结构，但 **逐格一致性、批量出图、失败重试** 仍偏手工。
- RAG/向量未强制要求；多模态深度主要依赖 **提示词工程 + 工作流参数 + 本地/远程 ComfyUI**。

---

## 二、目标（分阶段）

| 阶段 | 目标 | 验收 |
|------|------|------|
| **P0** | 单篇条漫/分镜：从 planner JSON → 逐格生成，格间风格与角色可控 | 同一 `article_id` 下连续 6 格视觉无明显「换人换画风」 |
| **P1** | ComfyUI 工作流 **文件级配置**：每 `content_format` × `image_style` 映射到可版本化的 JSON 工作流 | 换模型/换 LoRA 只需改配置与节点映射，不改 Python 分支爆炸 |
| **P2** | 批量队列：后台任务 + 缩略图预览 + 单格重绘 | 失败格可单独重试，不影响已接受格 |
| **P3** | （可选）本地 IP-Adapter / Reference 节点：参考图锁角色 | 医学插画角色一致性明显提升 |

---

## 三、架构要点

1. **编排层**（MedComm 后端）  
   - 维护：`workflow_id` → 磁盘路径 / 内嵌模板版本号。  
   - 请求 Comfy：`/prompt` 提交 + `/history` 轮询或 WebSocket（以你当前 Comfy 版本为准）。  
   - 将 `visual_continuity_prompt`、`image_series_seed_base`、`panel_index` 注入到工作流约定节点（与设计方案中的 node id 配置一致）。

2. **资产层**  
   - 生成结果仍落现有 `images` 目录与 `GeneratedImage`（或扩展 metadata：格序号、workflow 版本、seed）。  
   - 编辑器用现有 `medcomm-image` 协议引用；需保证 **格序号与 article section 的 format_meta 对齐**。

3. **前端**  
   - 在条漫/分镜形式下增加「一键生成本页全部格」「仅生成缺图格」「打开 Comfy 预览（可选）」。  
   - 个人版无需审批流；保留 **单格重试** 即可。

---

## 四、任务拆解（建议顺序）

1. **配置契约**  
   - 冻结一版 `comfy_workflow.schema.json`（或 YAML）：含 `prompt_node_id`、`negative_node_id`、`seed_node_id`、可选 `reference_image_node_id`。  
   - 与 `ComfyUI工作流编排设计方案.md` 对齐字段命名。

2. **后端**  
   - `imagegen` 增加 `run_comfy_batch_for_article(article_id, section_ids | panel_indices)`：读 planner + 当前格 `scene_desc`，循环提交 Comfy。  
   - 统一错误码：超时 / Comfy 未启动 / 显存不足 → 前端可提示「降分辨率或关并发」。

3. **前端**  
   - `Article.vue` 或形式专用子组件：条漫 Tab 上展示每格状态（未生成 / 生成中 / 完成 / 失败）。  
   - 调用新批量 API；与「图示连贯性」折叠面板联动说明文案。

4. **质量与成本**  
   - 默认并发 1～2，避免个人显卡 OOM。  
   - 提供「草稿分辨率」与「导出分辨率」两档（若工作流支持）。

5. **后续**  
   - 与 **个人语料** 联动：可在语料中增加「画面禁忌」类 note（需扩展 `kind` 或走 `note`），注入到 image prompt builder（独立小块，不与正文语料混太长）。

---

## 五、非目标（本计划不展开）

- 医院多用户权限与审计。  
- 云端 GPU 池计费。  
- 自动版权检测（可另立轻量任务）。

---

## 六、里程碑建议

- **M1（1～2 周）**：P0 单形式端到端 + 配置化 node 映射。  
- **M2（2～3 周）**：P1 工作流多模板 + 版本号写入生成元数据。  
- **M3（按需）**：P2 队列与单格重试；P3 Reference 一致性。
