# 安全机制完整性检查

对照 Schola v4 规范：LOCAL_API_KEY 本机隔离 + Keychain 统一存取。

---

## 规范要求

### Keychain 存取约定

| Service | Account | 用途 |
|---------|---------|------|
| `LinScio MedComm` | `user@email.com` | 门户 token |
| `LinScio MedComm - Keys` | `openai` | OpenAI API Key |
| `LinScio MedComm - Keys` | `siliconflow` | 硅基流动 API Key |
| `LinScio MedComm - Keys` | `deepseek` | DeepSeek API Key |

### LOCAL_API_KEY

- 本机隔离：每次启动生成 UUID，注入后端环境
- 用途：本地请求鉴权 / 仅接受本机客户端

---

## 当前实现

### LOCAL_API_KEY

| 项目 | 状态 | 说明 |
|------|------|------|
| 启动时生成 UUID | ✅ | `main.js` 中 `crypto.randomUUID()` |
| 注入后端 env | ✅ | `LINSCIO_LOCAL_API_KEY` 传入子进程 |
| 后端校验请求 | ❌ | 后端未使用该变量做鉴权 |

### Keychain

| 规范 | 当前 | 缺口 |
|------|------|------|
| Service `LinScio MedComm` | `com.linscio.medcomm` | ⚠️ 使用 bundle-id 风格，非显示名称 |
| Service `LinScio MedComm - Keys` | 无独立 Service | ❌ 未拆分 Keys 服务 |
| Account `user@email.com` → 门户 token | `USER_TOKEN` / `AUTH_TOKEN` | ⚠️ 账户名不一致 |
| Account `openai` | `OPENAI_API_KEY` | ⚠️ 使用 env 名作 account |
| Account `siliconflow` | 无 | ❌ 缺失 |
| Account `deepseek` | 无 | ❌ 缺失 |

### 前端 Settings

| 规范 | 当前 | 缺口 |
|------|------|------|
| OpenAI API Key | ✅ | 已支持 |
| Anthropic API Key | ✅ | 规范未要求，可保留 |
| 硅基流动 API Key | ❌ | 缺失 |
| DeepSeek API Key | ❌ | 缺失 |
| 门户 token / user@email.com | ❌ | 无 UI（单机模式可暂无） |

### 后端 Key 映射

`getApiKeysForBackend()` 当前注入：

- `OPENAI_API_KEY`
- `ANTHROPIC_API_KEY`
- `USER_TOKEN`
- `AUTH_TOKEN`

后端实际使用的 env：

- `OPENAI_API_KEY` ✅
- `ANTHROPIC_API_KEY` ✅
- `SILICONFLOW_API_KEY` ✅
- `DEEPSEEK_API_KEY` ✅

## 汇总：需补齐项（已实现）

| 优先级 | 类别 | 缺口 | 状态 |
|--------|------|------|------|
| P1 | Keychain | 使用双 Service：`LinScio MedComm`（门户 token）与 `LinScio MedComm - Keys`（API Key） | ✅ |
