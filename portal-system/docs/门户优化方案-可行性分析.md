## 二、门户 · 管理后台 · 镜像仓库 · 软件系统 — 授权衔接说明

以下说明「门户网站、管理后台、镜像仓库」三者如何通过**授权码（LicenseKey）**衔接，以及授权如何延续到**软件系统（linscio-ai）**的拉取与运行，便于在优化方案中复用同一套授权体系。

### 2.1 三端角色与数据流总览

```
┌─────────────────┐     ┌─────────────────┐     ┌─────────────────────────┐
│   管理后台       │     │   门户网站       │     │   镜像仓库（Registry）    │
│   (admin)       │     │   (portal)      │     │   + registry-proxy       │
├─────────────────┤     ├─────────────────┤     ├─────────────────────────┤
│ · 生成授权码     │────▶│ · 用户注册/登录   │     │ · 存镜像 (linscio-ai 等)  │
│ · 套餐/周期配置   │     │ · 用户输入授权码  │     │ · 拉取前校验：           │
│ · 作废/延期授权码 │     │ · 激活 → 获得    │────▶│   - 用户名/密码(htpasswd) │
│ · 查看用户/机器  │     │   Registry 凭证  │     │   - 授权未作废、未过期   │
└─────────────────┘     └─────────────────┘     │   - 可拉取 Tag/次数      │
        │                         │              └────────────┬────────────┘
        │                         │                           │
        │                         │              ┌────────────▼────────────┐
        │                         │              │  软件系统 (linscio-ai)   │
        └────────────────────────┼──────────────│  · docker pull 用凭证    │
                                  │              │  · 运行时用心跳带授权码   │
                                  └──────────────└─────────────────────────┘
```

**核心衔接点**：授权码（`LicenseKey`）在管理后台生成，在门户被用户「激活」后，既用于生成**镜像拉取凭证**（Registry 用户名/密码），又用于**软件运行时心跳**（授权码 + 机器 ID）。镜像仓库侧不直接存授权码，只认「用户名」；proxy 通过「用户名 → 凭证 → 授权码」反查套餐与作废状态。

### 2.2 授权管理衔接（三步闭环）

**管理后台：生成授权码**

- 管理员在「授权码管理」中按套餐、周期批量生成授权码，写入表 `license_keys`（`code`、`plan_id`、`period_id`、`expires_at`、`machine_limit`、`pull_limit_monthly`、`module_mask` 等）。
- 此时授权码尚未与任何用户绑定（`is_used=False`，`assigned_to=null`），镜像仓库中也没有对应该授权码的账号；授权码只是「待激活」记录，决定激活后能拉哪些 Tag、多少台机器等。

**门户网站：用户注册 + 激活授权码 → 获得镜像拉取凭证**

- 客户在门户注册/登录，在用户中心输入授权码并点击「立即激活」，调用 `POST /api/v1/user/activate`。
- 后端：校验授权码 → 将授权码与当前用户绑定（`assigned_to`、`is_used`、`activated_at`）→ 按套餐生成 Registry 用户名与密码 → 写入 **htpasswd** 与表 **registry_credentials**（`user_id`、**license_id**、`registry_username`、`allowed_image_tags`、`expires_at`）。
- 用户获得「Registry 地址、用户名、密码」，用于 `docker login` 和 `docker pull`。**门户激活 = 为该用户创建「绑定到该授权码」的 Registry 账号，并把该账号与授权码的套餐/作废状态挂钩。**

**镜像仓库：拉取时校验「凭证 ↔ 授权码」**

- 客户使用门户提供的用户名/密码做 `docker login` 与 `docker pull`；请求先到 **registry-proxy**。
- proxy：从 Authorization 得到**用户名** → 查 **registry_credentials** 得 `license_id`、`allowed_image_tags` → 查 **license_keys** 得 `is_revoked`、`pull_limit_monthly` → 作废/超限/Tag 不符则 403，否则转发并扣减拉取次数。
- 镜像仓库本身不存授权码，只认 htpasswd；**授权码的约束通过 registry_credentials 的 license_id 在 proxy 里生效。**

### 2.3 软件系统侧的衔接（拉取 + 运行）

- **拉取镜像**：客户用门户下发的 **Registry 凭证**（用户名/密码）做 `docker login` 与 `docker pull`；同一授权码对应唯一凭证，proxy 按该授权码的套餐与状态控制拉取。
- **运行时心跳**：linscio-ai 定期调用 `POST /api/v1/license/heartbeat`，携带**授权码明文**、`machine_id`、`instance_id`。后端用授权码查 **license_keys**，校验作废/过期/机器数/并发数，更新 **machine_bindings**、**instance_heartbeats**，返回 module_mask、quota 等。**同一授权码**既决定「谁能用哪个 Registry 账号拉镜像」，又决定「实例能否继续跑、能用哪些模块」。

### 2.4 数据关系小结

| 表/组件 | 作用 | 与授权码的衔接 |
|---------|------|----------------|
| **license_keys** | 授权码主表（管理后台生成） | 中心实体；portal 激活时写 assigned_to/activated_at；admin 作废/延期改 is_revoked/expires_at；proxy 与 heartbeat 都查此表 |
| **registry_credentials** | 门户用户 ↔ Registry 账号 | 一条记录对应一个用户+一个授权码（license_id）；激活时创建，凭证写入 htpasswd；proxy 用 registry_username 反查 license_id |
| **htpasswd** | Registry 认证 | 仅存「用户名+密码」；作废授权时从 htpasswd 删除对应用户名 |
| **machine_bindings** | 授权码 ↔ 机器 | 心跳时创建/更新；同一授权码可绑多台机器（受 machine_limit 限制） |
| **instance_heartbeats** | 授权码 ↔ 运行实例 | 心跳时创建/更新；用于并发数控制与在线统计 |

**一句话**：管理后台负责「授权码从无到有」；门户负责「授权码从码到人、到 Registry 凭证」；镜像仓库 + proxy 负责「只认凭证、按凭证背后的授权码做拉取控制」；软件系统用同一授权码做「拉取时的凭证」和「运行时的心跳」，在拉取与运行两端都与同一套授权衔接。

### 2.5 端到端流程（客户视角）

1. 管理员在管理后台生成授权码并发放给客户。  
2. 客户在门户注册并登录，在用户中心输入授权码并激活 → 获得 Registry 地址、用户名、密码。  
3. 客户在服务器上执行 `docker login` + `docker pull`（用上述凭证）→ registry-proxy 按该授权码的套餐与状态放行。  
4. 客户启动 linscio-ai 时配置**同一授权码**；应用定期调心跳 → 后端根据该授权码校验并返回模块与配额。  
5. 若管理员在后台作废或延期该授权码：拉取侧该 Registry 用户名从 htpasswd 删除或 proxy 查得 is_revoked → 无法再 pull；运行侧下次心跳返回 403 或 license_expired → 应用可拒绝继续运行或提示续费。

## 三、分项可行性说明

### 3.1 授权码验证接口（方案 3.1，P0）

**方案要求**：install.sh 调用；接收授权码 + 机器标识；校验签名与有效期；记录绑定机器、首次激活时间、客户端 IP；返回验证结果。

**现有能力**：

- **校验**：`api/app/services/license_service.py` 中已有 `validate_license_format`、`lookup_license_activate_status`；`LicenseKey` 含 `expires_at`、`is_revoked`、`activated_at`。
- **机器记录**：`MachineBinding`（license_id, machine_id, user_id, last_heartbeat）；当前由「用户中心激活」或「心跳」创建。
- **差异**：当前「激活」仅在门户完成（`POST /api/v1/user/activate`，需登录），install.sh 无法使用。

**可行方案**：

- 新增 **公开** 接口，例如 `POST /api/v1/public/license/verify`（或放在 `license` 下但无需 Token）：
  - 入参：`license_code`、`machine_id`；客户端 IP 从 `request.client.host` 或 `X-Forwarded-For` 取。
  - 逻辑：格式校验 → 查库 → 校验未作废、未过期 → 若未激活则置 `is_used=True`、`activated_at=now`（可选：`assigned_to` 留空或专用「安装激活」用户）→ 创建/更新 `MachineBinding`，记录 IP（可新加字段或写入 `OperationLog`）。
  - 返回：`{ "status": "ok" | "expired" | "revoked", "message": "..." }`，便于 install.sh 分支处理。
- **注意**：与现有「门户激活」的关系需约定：同一授权码可先被 install.sh 验证（仅记机器），再在门户被用户「认领」绑定账号；或规定「仅一种激活路径」。建议采用「install 仅验证+记机器，门户激活可绑定用户并生成 Registry 凭证」的双路径，两者共用 `LicenseKey`/`MachineBinding`。

**工作量**：约 0.5～1 天（接口 + 测试 + 文档）。

### 3.2 心跳验证接口（方案 3.2，P0）

**方案要求**：运行时每 24 小时调用；返回 ok / expired / revoked；系统据此决定是否拒绝启动。

**现有能力**：

- `POST /api/v1/license/heartbeat` 已实现：
  - 入参：`license_key`、`machine_id`、`instance_id`。
  - 校验：授权码存在、已激活、未作废、未过期（过期时仍 200，带 `license_expired=True`）；机器数/并发数校验。
  - 更新：`InstanceHeartbeat`、`MachineBinding.last_heartbeat`。

**可行方案**：

- 接口保持不变；在**产品文档**中明确约定：
  - HTTP 200 + `license_expired: false` → 视为 `ok`，继续运行。
  - HTTP 200 + `license_expired: true` → 视为 `expired`，记录警告，下次重启拒绝启动。
  - HTTP 403（授权已作废 / 无效等）→ 视为 `revoked` 或无效，记录警告，下次重启拒绝启动。
- 若希望与方案完全一致，可在响应中增加显式字段 `status: "ok" | "expired" | "revoked"`，便于脚本解析（可选，约 0.5 天）。

**工作量**：0～0.5 天（文档必做；可选字段按需）。

### 3.3 下载链接生成接口（方案 3.3，P0）

**方案要求**：前端传入授权码；校验合法性；查当前最新可下载版本；生成 COS 预签名 URL（2 小时）；记录下载日志；返回下载链接与安装文档地址。

**现有能力**：

- 无版本、无 COS、无下载日志。

**可行方案**：

- **存储与配置**：
  - 腾讯云 COS（F2）：私有 Bucket `linscio-registry-1363203425`（地域 `ap-hongkong`），目录按方案 `releases/`、`docs/`；在 API 环境变量中配置 `COS_SECRET_ID`、`COS_SECRET_KEY`、`COS_BUCKET`、`COS_REGION`。
  - 数据库：新增表 `release_versions`（如：id, version, file_key, file_size, release_notes, is_current, created_at）；可选表 `download_logs`（license_id, version, ip, user_agent, created_at）。
- **接口**：例如 `POST /api/v1/public/download/presign-url`（或 `/api/v1/download/generate-link`）：
  - 入参：`license_code`。
  - 逻辑：校验授权码（格式 + 未作废 + 未过期，可与 verify 复用）；查询 `release_versions` 中 `is_current=True` 的一条；用 COS SDK 生成预签名 URL（2 小时）；写 `download_logs`；返回 `{ download_url, expires_in_seconds, version, doc_url }`。
- **依赖**：`cos-python-sdk-v5` 或 `tencentcloud-sdk-python-cos`；需在 `api/requirements.txt` 增加并部署。

**工作量**：约 1～1.5 天（表设计 + 接口 + COS 集成 + 简单测试）。

## 四、数据与接口设计建议

### 4.1 建议新增表

- **release_versions**  
  - id, version, file_key（COS 路径）, file_size, release_notes, is_current, created_at  
  - 用于「当前可下载版本」及后台版本管理。
- **download_logs**  
  - id, license_id, version, client_ip, user_agent, created_at  
  - 用于下载记录与运营审计（方案 3.3、4.1）。

### 4.2 建议新增/扩展接口（汇总）

| 接口 | 方法 | 说明 |
|------|------|------|
| `/api/v1/public/license/verify` | POST | 授权码+机器验证，记录绑定与首次激活，返回 status（ok/expired/revoked） |
| `/api/v1/public/download/presign-url` 或 `/api/v1/download/generate-link` | POST | 授权码校验 + 当前版本 + COS 预签名 URL + 下载日志 |
| `/api/v1/public/docs/install-guide` | GET | 可选；返回安装文档 Markdown 或 HTML |
| `/api/v1/admin/release-versions` | GET/POST/PATCH | 版本列表、新建、设置当前版本 |
| `/api/v1/admin/download-logs` | GET | 下载记录分页查询 |

### 4.3 与现有模型的关系

- **LicenseKey**：不改变主结构；`activated_at`、`is_used` 可由 install.sh 的 verify 首次写入（若采用「安装即可激活」策略）。
- **MachineBinding**：继续用于「授权码 ↔ 机器」；verify 与 heartbeat 均可创建/更新。
- **RegistryCredential**：仅门户「用户激活」时创建；install.sh 仅验证+机器绑定，不生成镜像拉取凭证，与方案「安装包分发」场景一致。
- **OperationLog**：可选记录 verify 的 IP/时间，便于审计。

## 六、风险与注意点

1. **双激活路径**：install.sh 验证与门户用户激活并存时，需约定「未绑定用户」的授权码是否允许下载、是否允许后续被某用户认领；建议在业务文档与接口约定中写清。
2. **安全**：verify、presign-url 为公开接口，需做限流与防刷（如按 IP/授权码频率）；下载日志注意脱敏与合规。
3. **可用性**：授权与下载接口建议单独监控与告警；install.sh 超时与错误提示按方案 6.2 落实。
