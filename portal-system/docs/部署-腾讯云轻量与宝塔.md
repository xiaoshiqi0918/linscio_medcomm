# LinScio AI · 聆思恪 门户系统 · 腾讯云轻量 + 宝塔部署说明

适用环境：腾讯云轻量应用服务器（4 vCPU · 4 GB 内存）、Ubuntu 24.04 LTS、宝塔面板、域名 linscio.com.cn。

## 域名规划

| 子域名 | 用途 | 对外说明 |
|--------|------|----------|
| `www.linscio.com.cn` | 门户网站主站 | 对外主入口 |
| `linscio.com.cn` | 根域 | 301 跳转至 www |
| `admin.linscio.com.cn` | 管理后台 | 仅内部运营使用 |
| `registry.linscio.com.cn` | Docker 镜像仓库 API | 用户 docker login / pull |
| `hub.linscio.com.cn` | 镜像仓库 Web UI | 管理员查看镜像 |

## 部署架构

### 3.1 架构总览

```
[用户浏览器 / Docker Client]
            │  HTTPS (443)
            ▼
    ┌──────────────┐
    │  宝塔 Nginx   │  ← SSL 终止 + 反向代理
    └──────┬───────┘
           │ 内网代理 (127.0.0.1)
    ┌──────┴────────────────────────────────┐
    │         Docker Compose 网络            │
    │                                        │
    │  ┌──────────┐    ┌─────────────────┐   │
    │  │ portal-  │    │   linscio-api    │   │
    │  │ frontend │    │  (FastAPI 容器)  │   │
    │  │(Nginx静态)│    │   :8000         │   │
    │  └──────────┘    └────────┬────────┘   │
    │                           │            │
    │  ┌──────────┐    ┌────────▼────────┐   │
    │  │  admin-  │    │   linscio-db   │   │
    │  │ frontend │    │(PostgreSQL容器) │   │
    │  │(Nginx静态)│    │   :5432        │   │
    │  └──────────┘    └─────────────────┘   │
    │                                        │
    │  ┌──────────┐    ┌─────────────────┐   │
    │  │ registry │    │  registry-ui     │   │
    │  │  :5000   │    │    :5001         │   │
    │  │(内网仅暴露)   │  │(代理到 registry)   │   │
    │  └──────────┘    └─────────────────┘   │
    │  ┌─────────────────────────────────┐   │
    │  │ linscio-registry-proxy :5000     │   │  ← 对外 5000：按 Tag 白名单校验后转发
    │  └─────────────────────────────────┘   │
    └────────────────────────────────────────┘
```

### 3.2 各组件部署方式

| 组件 | 部署方式 | 端口（内网） | 说明 |
|------|----------|--------------|------|
| 门户前端 Vue | Docker Nginx 静态容器 | 3000 | 镜像内构建产物，Nginx 托管 |
| 管理后台 Vue | Docker Nginx 静态容器 | 3001 | 镜像内构建产物，Nginx 托管 |
| FastAPI 后端 | Docker 容器 | 8000 | 4 workers |
| PostgreSQL | Docker 容器 | 5432 | 数据持久化挂载 |
| **Registry 拉取代理** | Docker 容器（同 API 镜像） | **5000** | 按 allowed_image_tags 校验 Tag，再转发到 Registry；**对外暴露 5000** |
| Docker Registry | Docker 容器 | 5000（仅容器间） | 仅内网 expose，不直接对外 |
| Registry UI | Docker 容器 | 5001 | 依赖 Registry 容器，直连 Registry |
| 宝塔 Nginx | 宝塔原生安装 | 80/443 | 统一入口，SSL 终止 |

### 3.3 端口规划

| 端口 | 用途 | 是否对外暴露 |
|------|------|----------------|
| 80 | HTTP（建议强制跳转 HTTPS） | ✅ 是 |
| 443 | HTTPS 统一入口 | ✅ 是 |
| 3000 | 门户前端容器（内网） | ❌ 否 |
| 3001 | 管理后台容器（内网） | ❌ 否 |
| 8000 | FastAPI 容器（内网） | ❌ 否 |
| 5432 | PostgreSQL 容器（内网） | ❌ 否 |
| 5000 | Registry 拉取代理（Tag 白名单校验） | ❌ 否（仅 127.0.0.1，由 Nginx 代理） |
| 5001 | Registry UI 容器（内网） | ❌ 否 |
| 8888 | 宝塔面板 | ⚠️ 建议在宝塔中改为随机端口 |

## 2. 服务器准备

### 2.1 安装 Docker 与 Docker Compose

若未安装：

```bash
curl -fsSL https://get.docker.com | sh
sudo usermod -aG docker $USER
# 登出再登录后
docker compose version
```

**国内服务器必做：配置 Docker 镜像加速**  
若 `docker compose build` 时出现 `failed to resolve source metadata for docker.io/library/node:20-alpine ... i/o timeout`，说明无法访问 Docker Hub，需先配置镜像加速。若使用腾讯云镜像 `mirror.ccs.tencentyun.com` 拉取时报 **500 Internal Server Error**，可改用下方其他镜像。

```bash
sudo mkdir -p /etc/docker
# 任选其一或配置多个备用（推荐先试 docker.1ms.run）
sudo tee /etc/docker/daemon.json <<'EOF'
{
  "registry-mirrors": [
    "https://docker.1ms.run",
    "https://docker.xuanyuan.me"
  ]
}
EOF
sudo systemctl daemon-reload
sudo systemctl restart docker
```

腾讯云专用：`https://mirror.ccs.tencentyun.com`。阿里云/华为云等请用各自镜像服务提供的地址。

### 2.2 目录与权限

```bash
sudo mkdir -p /www/wwwroot/linscio
sudo chown -R $USER:$USER /www/wwwroot/linscio
cd /www/wwwroot/linscio
```

将本仓库 `portal-system` 目录上传到该路径（或 git clone），保证存在 `docker-compose.yml`、`api/`、`portal/`、`admin/`、`registry/`、`registry-ui/` 等。

## 4. 首次启动前准备

### 4.1 Registry 认证文件

```bash
./scripts/create-registry-htpasswd.sh
# 或自定义：./scripts/create-registry-htpasswd.sh 用户名 密码
```

### 4.2 构建并启动所有服务

```bash
docker compose build
docker compose up -d
```

### 4.3 创建管理员（并同步到 Registry 认证）

```bash
./scripts/create-admin.sh 管理员用户名 管理员密码
# 脚本会同步到 htpasswd 并提示重启 Registry；若已启动 Registry，再执行：
docker compose restart linscio-registry
```

### 4.4 恢复管理员（含镜像仓库）

当管理员账号或镜像仓库认证被误删、需重新恢复时，在 **portal-system 目录下**执行：

```bash
# 1. 确保 API、数据库已启动
docker compose up -d linscio-db linscio-api

# 2. 恢复管理员（默认 admin / admin123；可改为你的用户名和密码）
./scripts/create-admin.sh
# 或指定账号：./scripts/create-admin.sh 你的用户名 '你的密码'

# 3. 重启 Registry 使新认证生效
docker compose restart linscio-registry
```

- **数据库中已无该管理员**：上述脚本会重新创建并写入镜像仓库认证。
- **数据库中仍有该管理员、仅镜像仓库认证丢失**：脚本会跳过创建，但会**把该账号密码同步到 htpasswd**，执行后即可用同一账号登录管理后台与 Registry / Registry UI。
- **需要同时修改已有管理员的密码**：  
  `./scripts/create-admin.sh 用户名 新密码 update`  
  会更新数据库中的密码并同步到镜像仓库。

## 5. 宝塔 Nginx 反向代理

### 5.1 域名解析

按上文「域名规划」在域名服务商处添加 A 记录，将以下主机记录解析到服务器公网 IP（1.15.136.213）：

| 主机记录 | 记录类型 | 解析结果 |
|----------|----------|----------|
| @        | A        | 1.15.136.213 |
| www      | A        | 1.15.136.213 |
| admin    | A        | 1.15.136.213 |
| registry | A        | 1.15.136.213 |
| hub      | A        | 1.15.136.213 |

### 5.2 在宝塔中配置站点与 Nginx

- 在宝塔「网站」中为 **linscio.com.cn**（根域）、**www.linscio.com.cn**、**admin.linscio.com.cn**、**registry.linscio.com.cn**、**hub.linscio.com.cn** 分别添加站点（或按需使用「反向代理」）。
- 在对应站点的「设置」→「配置文件」中，将 `deploy/baota-nginx-linscio.conf` 中对应 server 块内容合并进去，并确保：
  - `proxy_pass` 指向 `http://127.0.0.1:端口`（门户 3000 / 管理后台 3001 / Registry 5000 / Hub 5001）。
  - 根域 linscio.com.cn 仅做 301 跳转到 www，无需 proxy_pass。
  - 若启用 HTTPS，在宝塔中为各域名申请 SSL 证书，证书路径会由宝塔自动写入。

也可将 `deploy/baota-nginx-linscio.conf` 整份内容放到宝塔「Nginx」→「配置修改」的 `include` 目录下，或拆成多个 `server` 并入现有配置。

### 5.3 端口与 HTTPS

- Compose 中服务仅绑定 **127.0.0.1**，对外由宝塔 Nginx 统一入口，无需在轻量服务器防火墙开放 3000/3001/8000/5000/5001；仅需开放 **80、443**（宝塔默认会配置）。
- 建议在宝塔各站点中开启 **「强制 HTTPS」**，使 HTTP(80) 自动 301 跳转到 HTTPS(443)。

## 7. 日常运维

- **重启全部服务**：`docker compose restart`
- **仅重启 API**：`docker compose restart linscio-api`

更多运维细节（健康检查、日志排障、升级流程、常见问题）见 [8. 运维细节](#8-运维细节)。

### 7.1 更新 / 重构系统时保留已注册用户（重要）

**原因**：用户、授权码等数据存放在 PostgreSQL 的 **命名卷 `postgres-data`** 中。若在「更新、重构」时执行了 **`docker compose down -v`**，会删除所有命名卷（含 `postgres-data`），导致**已注册用户、授权码、绑定关系等全部丢失**。

**正确做法**：

- **日常更新代码/镜像**：只重新构建并启动，**不要**加 `-v`：
  ```bash
  cd /www/wwwroot/linscio/portal-system
  docker compose build
  docker compose up -d
  ```
- **仅重启服务**：`docker compose restart` 或 `docker compose up -d`（不删容器、不删卷）。
- **必须完全 teardown 时**：先 [备份数据库](#72-数据备份)，再 `docker compose down`（**不要**加 `-v`），这样卷会保留；下次 `docker compose up -d` 数据仍在。
- **仅需清理 Registry 镜像数据时**：可单独删除对应卷，例如：
  ```bash
  docker compose stop linscio-registry
  docker volume rm portal-system_registry-data  # 仅删 registry 卷，不影响 postgres-data
  docker compose up -d
  ```

**总结**：只要**不**执行 `docker compose down -v`，且未更换服务器或未丢弃 `postgres-data` 卷，更新、重构后已注册用户不会丢失。

### 7.2 数据备份

建议配置每日数据库备份，详见 [数据库设计.md](数据库设计.md#84-数据备份策略)。可使用项目提供的脚本（需在 portal-system 目录下执行）：

```bash
mkdir -p /www/wwwroot/linscio/data/backups
cd /www/wwwroot/linscio/portal-system && ./scripts/backup-db.sh
```

在宝塔「计划任务」中添加 Shell 脚本，每日 3:00 执行上述命令即可；备份目录与保留天数可通过环境变量 `BACKUP_ROOT`、`RETENTION_DAYS` 调整。

### 7.3 从旧版升级：补全表结构（license_keys、registry_credentials）

若数据库是早期版本创建的，部分表可能缺少新列，会导致管理后台「数据总览」报错，例如：
- `column license_keys.plan_id does not exist`
- `column registry_credentials.pull_count_this_month does not exist`

**处理步骤**（在 portal-system 目录下执行一次即可）：

```bash
chmod +x scripts/run-migrate-license-keys.sh
./scripts/run-migrate-license-keys.sh
```

脚本会依次执行：
- `migrate_license_keys_add_columns.sql`：为 `license_keys` 补全 plan_id、period_id、module_mask 等列
- `migrate_registry_credentials_add_columns.sql`：为 `registry_credentials` 补全 license_id、allowed_image_tags、pull_count_this_month、pull_count_reset_at、expires_at

或手动执行 SQL（需先确保 `plans`、`billing_periods` 表已存在）：

```bash
docker compose exec -T linscio-db psql -U linscio_user -d linscio_portal < scripts/migrate_license_keys_add_columns.sql
docker compose exec -T linscio-db psql -U linscio_user -d linscio_portal < scripts/migrate_registry_credentials_add_columns.sql
```

执行完成后重启 API：`docker compose restart linscio-api`，再刷新管理后台数据总览。

## 8. 运维细节

### 8.1 健康检查

| 服务 | 检查地址 | 说明 |
|------|----------|------|
| API | `http://127.0.0.1:8000/health` | 返回 `{"status":"ok"}` 表示正常 |
| Registry 拉取代理 | `http://127.0.0.1:5000/health` | 返回 `{"status":"ok","service":"registry-proxy"}` |
| PostgreSQL | 容器内 `pg_isready -U linscio_user -d linscio_portal` | compose 中 linscio-db 已配置 healthcheck |

可在宝塔「计划任务」或监控系统中对 `http://127.0.0.1:8000/health` 做定时请求，失败时告警或自动重启：

```bash
docker compose restart linscio-api
```

### 8.2 日志与排障

- **查看所有服务日志**：`docker compose logs -f`
- **单服务最近 200 行**：`docker compose logs --tail=200 linscio-api`
- **仅错误级别**：`docker compose logs linscio-api 2>&1 | grep -i error`

常见排障：

| 现象 | 可能原因 | 处理 |
|------|----------|------|
| **请求返回 502（含浏览器 "Request failed with status code 502"）** | 上游服务未启动、崩溃或 Nginx 代理配置错误 | 见下方 [502 排查](#502-排查) |
| 门户/管理后台 502 | API 未启动或崩溃 | `docker compose ps` 查看 linscio-api 状态；`docker compose logs linscio-api` 查看报错；重启 `docker compose restart linscio-api` |
| 管理后台「请先登录」/ 401 | token 过期或未登录 | 重新登录；检查浏览器 localStorage 中 admin_token |
| 用户 docker pull 返回 403 | 套餐不允许拉取该 Tag | 属预期：按 Tag 白名单管控；让用户使用门户「用户中心」中展示的可拉取 Tag 示例 |
| 用户 docker pull 返回 502 | 拉取代理连不上内网 Registry | 确认 linscio-registry 与 linscio-registry-proxy 均运行；`docker compose logs linscio-registry-proxy` 查看是否有 UNAVAILABLE |
| Registry 无法启动 | htpasswd 不存在或格式错误 | 执行 `./scripts/create-registry-htpasswd.sh` 后 `docker compose restart linscio-registry` |
| 数据总览/授权码列表报 500 | 数据库缺列（旧版升级） | 按 [7.3](#73-从旧版升级补全表结构license_keysregistry_credentials) 执行迁移脚本并重启 API |

#### 502 排查

1. **确认是哪个请求 502**  
   - 浏览器 F12 → Network，看失败请求的 URL：  
     - 若为 `/api/v1/...`，多半是 **API（linscio-api）** 未响应或 Nginx 代理到 API 的配置有问题。  
     - 若为 `registry.linscio.com.cn` 的拉取（如 `/v2/...`），则是 **registry-proxy** 或其后端 Registry 不可用。

2. **API 导致 502 时**  
   ```bash
   docker compose ps
   docker compose logs --tail=100 linscio-api
   curl -s http://127.0.0.1:8000/health
   ```  
   若容器退出或 `/health` 无响应：先 `docker compose up -d linscio-api`，再根据日志修错（如缺环境变量、数据库连不上等）。

3. **registry 拉取 502 时**  
   ```bash
   docker compose ps linscio-registry linscio-registry-proxy
   curl -s http://127.0.0.1:5000/health
   docker compose logs --tail=50 linscio-registry-proxy
   ```  
   确保 linscio-registry 与 linscio-registry-proxy 都在运行；若 proxy 日志里出现 upstream 连接错误，检查 `REGISTRY_UPSTREAM_URL` 与网络。

4. **宝塔 Nginx 代理**  
   若本机 `curl http://127.0.0.1:8000/health` 正常但浏览器仍 502，检查宝塔里对应站点的「反向代理」是否指向正确端口（如 API 为 8000、registry 为 5000），以及上游是否勾选「备用」、超时是否过短。

建议对 `postgres-data` 卷所在磁盘做空间监控；日志过多时可配置 Docker 日志驱动与轮转（如 `json-file` + `max-size=50m`）。

### 8.3 升级发布流程

1. **备份**：执行 [7.2 数据备份](#72-数据备份)。
2. **拉代码**：`cd /www/wwwroot/linscio/portal-system && git pull`（或上传新文件）。
3. **构建并启动**：`docker compose build && docker compose up -d`。**不要**使用 `docker compose down -v`。
4. **验证**：访问 `https://www.linscio.com.cn`、`https://admin.linscio.com.cn`，检查 API `/health`。
5. 若有数据库结构变更，按文档执行迁移脚本后 `docker compose restart linscio-api`。

### 8.4 常见问题速查

| 问题 | 处理 |
|------|------|
| 更新后已注册用户消失 | 多半执行过 `down -v` 导致卷被删；只能从备份恢复；日常更新勿用 `-v`，见 [7.1](#71-更新--重构系统时保留已注册用户重要) |
| 管理员忘记密码 / 管理员被删 | 使用 [4.4 恢复管理员](#44-恢复管理员含镜像仓库)；改密用 `create-admin.sh 用户名 新密码 update` |
| 授权码列表/数据总览报错缺列 | 执行 [7.3](#73-从旧版升级补全表结构license_keysregistry_credentials) 迁移并重启 API |
| docker pull 提示 403 DENIED | 拉取代理按套餐 Tag 校验；用户需使用其套餐允许的镜像 Tag（见门户用户中心「可拉取镜像 Tag 示例」） |
| Registry UI 无法删镜像 | 确认 `registry/config.yml` 中 `storage.delete.enabled: true` 且 Registry 容器已挂载该配置 |

### 8.5 可选：Registry v2 Token Auth

当前默认使用 **htpasswd + 拉取代理**（按 Tag 白名单校验）。若需启用 Registry 标准 Token 认证（docker login 时由 API 签发 JWT、Registry 验签放行）：

1. **API 已提供**：GET `/api/v1/registry/token`（Query：service、scope；Header：Authorization: Basic）。校验 portal 凭证后签发 JWT（HS256，1 小时有效）。
2. **暴露 Token URL**：使 `registry.linscio.com.cn` 的 `/token` 反向代理到 API，例如 Nginx 增加 `location /token { proxy_pass http://127.0.0.1:8000/api/v1/registry/token; ... }`，或使用独立域名指向 API 并配置 CORS。
3. **Registry 配置**：在 `registry/config.yml` 中启用 token auth，例如：
   - `auth.token.realm`: `https://registry.linscio.com.cn/token`（或实际 Token 服务 URL）
   - `auth.token.service`: `registry`（或当前 registry 服务名）
   - `auth.token.issuer`: `LinScio AI Registry`（与 API 配置 `REGISTRY_TOKEN_ISSUER` 一致）
   - 若 Registry 需自行验签：需使用 RS256 公钥并配置 `auth.token.rootcertbundle`；当前 API 使用 HS256，可与拉取代理配合（代理验 JWT 后转发）。

启用后，docker login 会收到 401 并跳转至 realm URL 获取 Token，再持 Token 访问 Registry。**按 Tag 的管控**仍建议由拉取代理完成（Token 内仅含仓库级 access）。

## 9. 可选：内存限制（4 GB 机器）

若需为容器设置内存上限，可使用项目自带的 `docker-compose.prod.yml` 覆盖资源限制后启动：

```bash
docker compose -f docker-compose.yml -f docker-compose.prod.yml up -d
```

各服务默认上限见 `docker-compose.prod.yml`（如 DB 384M、API 256M 等），可按需调整。不使用时直接 `docker compose up -d` 即可。
