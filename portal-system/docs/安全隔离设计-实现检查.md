# 安全隔离设计 — 实现检查

对照 8.1～8.4 的 JWT 体系、接口访问控制、htpasswd 安全与 CORS，核对实现与文档的一致性。

---

## 8.1 JWT 三套独立体系

| 使用方 | 文档 | 实现 |
|--------|------|------|
| 门户用户 | PORTAL_JWT_SECRET，7 天 | ✅ 使用 `JWT_SECRET_KEY`（可视为门户 Secret），`JWT_EXPIRE_MINUTES=10080`（7 天） |
| 管理员 | ADMIN_JWT_SECRET，8 小时 | ✅ `ADMIN_JWT_SECRET = JWT_SECRET_KEY + "_admin"`；`JWT_EXPIRE_MINUTES_ADMIN=480`（8 小时），独立有效期 |
| Registry Token | REGISTRY_JWT_SECRET，1 小时 | ✅ 可选配置 `REGISTRY_JWT_SECRET`，未设则回退 `JWT_SECRET_KEY`；签发时 `expires_in=3600`（1 小时） |

生产建议：在 .env 中单独配置 `REGISTRY_JWT_SECRET`，与门户、管理 JWT 完全隔离。

## 8.2 接口访问控制

| 文档描述 | 实现 |
|----------|------|
| 门户 /api/v1/user/* → 验证 Portal JWT，只能操作自己的数据 | ✅ `get_current_user` 依赖门户 JWT，按 `user.id` 过滤 |
| 管理 /api/v1/admin/* → 验证 Admin JWT | ✅ `get_current_admin` 使用 `ADMIN_JWT_SECRET` 解码 |
| 验证管理员 scope（super/operator），operator 不能作废/删除 | ✅ `require_super`：仅 `scope="super"` 可调用作废授权、强制解绑机器；其他管理接口仍仅需 `get_current_admin` |
| Registry Token /api/v1/registry/token → Basic Auth，不需 JWT | ✅ 校验 Basic 用户名密码后签发 Token |
| 应用心跳 /api/v1/license/heartbeat → 校验 license_key，不需用户 JWT | ✅ 无 JWT 依赖，按 code + 库内有效性校验 |

## 8.3 htpasswd 文件操作安全

| 文档描述 | 实现 |
|----------|------|
| ① 用户激活时添加 | ✅ 激活流程中 `write_htpasswd(reg_username, reg_password)` |
| ② 管理员作废时删除 | ✅ 作废时 `remove_from_htpasswd(cred.registry_username)` |
| ③ 授权自然到期时定时清理 | ✅ 定时任务每 `HTPASSWD_CLEANUP_INTERVAL_SECONDS` 秒（默认 3600）查询已过期凭证并从 htpasswd 删除，并触发 SIGHUP |
| ④ 批量更新凭证时更新 | ✅ sync-modules 为每条凭证重新生成密码、写 htpasswd、更新 DB 加密密码，并触发 SIGHUP |
| 每次操作使用文件锁（fcntl） | ✅ `write_htpasswd` / `remove_from_htpasswd` 使用 `fcntl.flock(LOCK_EX)`，无 fcntl 时退化为无锁 |
| 写入后触发 Registry reload（SIGHUP） | ✅ 若配置 `REGISTRY_PID_FILE`（PID 文件路径），每次写/删 htpasswd 后向该进程发 SIGHUP |
| 文件权限 chmod 600 | 部署侧保证；`os.open(..., 0o600)` 在创建新文件时使用 0o600 |
| Registry 容器只读挂载 | 部署侧（如 docker-compose 或 K8s 中 htpasswd 卷只读挂给 Registry） |

## 8.4 跨域请求控制（CORS）

| 文档描述 | 实现 |
|----------|------|
| CORS 配置允许门户、管理后台域名 | ✅ `CORS_ORIGINS` 从配置读取（JSON 数组），默认含 localhost；生产可设为 `["https://www.linscio.com.cn","https://admin.linscio.com.cn"]` |
| Registry Token 接口不走浏览器，无需 CORS | ✅ 无单独 CORS 白名单，与 API 共用中间件即可 |

---

**结论**：8.1 JWT 三套体系（门户 7 天、管理 8 小时、Registry 独立 Secret + 1 小时）已实现；8.2 接口访问控制与 super 限制已实现；8.3 htpasswd 写入/删除与文件锁已实现，到期清理与 SIGHUP 为可选；8.4 CORS 以配置为准。生产环境建议设置 `REGISTRY_JWT_SECRET` 并在部署中保证 htpasswd 权限与 Registry 只读挂载。
