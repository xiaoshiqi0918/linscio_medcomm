## 五、Registry 权限控制（与架构图的差异）

架构图描述：

- **FastAPI Token 认证服务（Registry v2 Token Auth）**
- docker login 时**验证凭证**，**动态签发**镜像访问 **JWT Token**
- Token 内嵌**套餐权限 + 有效期**
- Docker Registry **按 Token 权限**放行/拒绝拉取

当前实现：

- **认证方式**：Docker Registry 使用 **htpasswd**（用户名/密码文件），由 FastAPI 在用户激活时写入；管理员通过脚本同步到同一 htpasswd。
- **权限控制**：**不**使用 Registry 自带的 Token Auth；在 Registry 前增加 **linscio-registry-proxy**（独立 ASGI 应用，与主 API 同镜像）：
  - 对外暴露 5000，转发请求到 Registry；
  - 对 `GET /v2/<name>/manifests/<ref>`（ref 为 Tag 时）根据 **Basic Auth 用户名** 查 `registry_credentials.allowed_image_tags`，不匹配则返回 403；
  - 不签发 JWT，不修改 Registry 的 auth 配置。

对照结论：

| 维度 | 架构图 | 当前实现 | 对齐情况 |
|------|--------|----------|----------|
| 权限数据来源 | 仅由 FastAPI/DB 提供 | ✅ 仅 FastAPI 查 DB（registry_proxy 查 `registry_credentials`） | ✅ 一致 |
| Registry 是否直连 DB | 否 | ✅ 否，Registry 不直连 DB | ✅ 一致 |
| 谁做「按权限放行」 | FastAPI Token 服务 | FastAPI 侧 **registry-proxy**（按 Tag 白名单） | ✅ 逻辑一致 |
| docker login 时 | Token 服务验证凭证并签发 JWT | **htpasswd** 验证；无 Token 签发 | ❌ 实现方式不同 |
| 拉取时校验 | Registry 验 Token | **Proxy** 验 Basic 用户 + DB 白名单 | ⚠️ 效果等价，实现路径不同 |

**总结**：  
- **管控数据只存一份、由 FastAPI 统一读写**：已满足；Registry 不直接操作数据库，权限由 FastAPI 侧代理（registry-proxy）依据 DB 做校验。  
- **「Token 认证服务」**：已实现 **GET /api/v1/registry/token**（Basic 校验后签发 JWT，HS256）；默认仍为 htpasswd + proxy。若需 Registry 直接验签，需配置 Registry auth.token（realm 指向本接口、rootcertbundle 等）。
