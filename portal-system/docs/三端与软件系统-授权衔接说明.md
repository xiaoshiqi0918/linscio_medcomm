# 门户 · 管理后台 · 镜像仓库 · 软件系统 — 授权衔接说明

本文说明「门户网站、管理后台、镜像仓库」三者如何通过**授权码（LicenseKey）**衔接，以及授权如何延续到**软件系统（linscio-ai）**的拉取与运行。

---

## 一、三端角色与数据流总览

```
┌─────────────────┐     ┌─────────────────┐     ┌─────────────────────────┐
│   管理后台       │     │   门户网站       │     │   镜像仓库（Registry）    │
│   (admin)       │     │   (portal)      │     │   + registry-proxy       │
├─────────────────┤     ├─────────────────┤     ├─────────────────────────┤
│ · 生成授权码     │────▶│ · 用户注册/登录   │     │ · 存镜像 (linscio-ai 等)  │
│ · 套餐/周期配置   │     │ · 用户输入授权码  │     │ · 拉取前校验：           │
│ · 作废/延期授权码 │     │ · 激活 → 获得    │────▶│   - 用户名/密码(htpasswd) │
│ · 查看用户/机器  │     │   Registry 凭证  │     │   - 授权未作废、未过期   │
└─────────────────┘     └─────────────────┘     │   - 可拉取 Tag/次数      │
        │                         │              └────────────┬────────────┘
        │                         │                           │
        │                         │              ┌────────────▼────────────┐
        │                         │              │  软件系统 (linscio-ai)   │
        └────────────────────────┼──────────────│  · docker pull 用凭证    │
                                  │              │  · 运行时用心跳带授权码   │
                                  └──────────────└─────────────────────────┘
```

**核心衔接点**：授权码（`LicenseKey`）在管理后台生成，在门户被用户「激活」后，既用于生成**镜像拉取凭证**（Registry 用户名/密码），又用于**软件运行时心跳**（授权码 + 机器 ID）。镜像仓库侧不直接存授权码，只认「用户名」；proxy 通过「用户名 → 凭证 → 授权码」反查套餐与作废状态。

## 二、授权管理衔接（三步闭环）

### 2.1 管理后台：生成授权码

- **谁**：管理员登录管理后台（`admin` 前端，`/api/v1/admin/*`）。
- **做什么**：
  - 在「授权码管理」中选择套餐（如专业版）、付费周期（月/季/年），批量生成授权码（如 `LS-PY-XXXX-XXXX-XX`）。
  - 授权码写入表 `license_keys`，包含：`code`、`plan_id`、`period_id`、`expires_at`、`machine_limit`、`concurrent_limit`、`pull_limit_monthly`、`module_mask` 等。
- **结果**：授权码尚未与任何用户绑定（`is_used=False`，`assigned_to=null`），仅存在于数据库中，可发放给客户。

**与镜像仓库的关系**：此时镜像仓库中还没有对应该授权码的账号；授权码只是「待激活」的一条记录，决定将来激活后能拉哪些 Tag、多少台机器等。

---

### 2.2 门户网站：用户注册 + 激活授权码 → 获得镜像拉取凭证

- **谁**：客户在门户（`portal` 前端）注册账号并登录。
- **做什么**：
  1. 注册/登录后进入「用户中心」；
  2. 输入管理员发放的**授权码**，点击「立即激活」；
  3. 调用 `POST /api/v1/user/activate`（需登录 Token）。
- **后端逻辑**（衔接核心）：
  1. 校验授权码格式、是否存在、未作废、未过期、可激活；
  2. 将授权码与当前用户绑定：`license_key.assigned_to = user.id`，`license_key.is_used = True`，`license_key.activated_at = now`；
  3. 按套餐生成 **Registry 用户名**（如 `pro_user_xxxxxxxx`）和随机**密码**；
  4. 将**用户名+密码**写入 **htpasswd**（`registry/auth/htpasswd`），供 Docker Registry 校验；
  5. 在表 **registry_credentials** 中写入一条记录：`user_id`、**license_id**、`registry_username`、加密后的 `registry_password_enc`、`allowed_image_tags`（由套餐决定）、`expires_at`（同授权码到期日）。
- **结果**：用户可在「用户中心」看到「Registry 登录凭证」：**地址、用户名、密码**，用于 `docker login` 和 `docker pull`。

**与镜像仓库的衔接**：  
- 镜像仓库存镜像（如 `linscio-ai:latest`），**不存授权码**，只做「用户名+密码」认证（htpasswd）。  
- **registry-proxy**（在拉取前）根据**用户名**查 `registry_credentials` → 得到 **license_id** → 再查 **license_keys** 得到：是否作废、可拉取 Tag 列表、月拉取上限等，据此放行或拒绝。  
因此：**门户激活 = 为这名用户创建「绑定到该授权码」的 Registry 账号，并把该账号与授权码的套餐/作废状态挂钩。**

---

### 2.3 镜像仓库：拉取时校验「凭证 ↔ 授权码」

- **谁**：客户在本机执行 `docker login <registry_url>`（使用门户提供的用户名/密码），再 `docker pull ...`。
- **请求路径**：请求先到 **registry-proxy**（如 1.15.136.213:5000），proxy 再转发到后端 Docker Registry（如 linscio-registry:5000）。
- **proxy 逻辑**（衔接核心）：
  1. 从请求头 `Authorization: Basic <base64(username:password)>` 得到**用户名**；
  2. 用该用户名查 **registry_credentials**，得到 `allowed_image_tags`、`license_id`；
  3. 通过 **license_id** 查 **license_keys**，得到 `is_revoked`、`pull_limit_monthly` 等；
  4. 若授权已作废 → 403；若请求的镜像 Tag 不在 `allowed_image_tags` 内 → 403；若本月拉取次数已达上限 → 403；
  5. 通过则转发到 Registry，并在拉取成功后扣减 `pull_count_this_month`、写 `pull_records`。

**总结**：镜像仓库本身不「认识」授权码，只认识 htpasswd 里的用户名；**授权码的约束（作废、套餐、拉取上限）通过 registry_credentials 的 license_id 在 proxy 里生效**。

## 三、软件系统侧的衔接（拉取 + 运行）

### 3.1 拉取镜像（与门户/镜像仓库的衔接）

- 客户在门户激活后拿到的是一组 **Registry 凭证**（地址 + 用户名 + 密码），不是授权码明文。
- 在部署机器上：
  - `docker login <REGISTRY_URL>`：输入上述**用户名和密码**；
  - `docker pull <REGISTRY_URL>/linscio-ai:xxx`：由 registry-proxy 按该用户对应的**授权码套餐**校验 Tag 与拉取次数。
- **衔接点**：同一授权码在门户激活后生成**唯一**的 registry 凭证；该凭证在 proxy 里永远关联到同一个 **license_id**，因此「授权码的套餐、作废、拉取上限」一致地作用在每次 pull 上。

### 3.2 运行时心跳（与门户/管理后台的衔接）

- 软件系统（linscio-ai）启动后，会定期（如每 24 小时）调用 **`POST /api/v1/license/heartbeat`**，携带：
  - **license_key**：授权码**明文**（如 `LS-PY-XXXX-XXXX-XX`）；
  - **machine_id**：本机标识；
  - **instance_id**：当前实例 ID。
- **后端逻辑**：
  1. 用 **license_key** 查 **license_keys**（且要求 `is_used=True`）；
  2. 若未找到或已作废 → 403；
  3. 若已过期 → 仍 200，但响应里 `license_expired=True`（由应用侧决定是否拒绝下次启动）；
  4. 校验机器数/并发数上限，必要时创建或更新 **machine_bindings**、**instance_heartbeats**；
  5. 返回 200 及套餐信息（如 module_mask、quota 等），供应用解锁功能、展示剩余次数等。

**衔接点**：  
- 心跳用的是**授权码**，不是 Registry 用户名。  
- 授权码在**门户**被用户激活后，`license_keys` 上就有 `assigned_to`、`activated_at`；**管理后台**作废或延期，改的是同一条 **license_keys**。  
- 因此：**同一授权码**既决定了「谁能用哪个 Registry 账号拉镜像」（通过 registry_credentials），又决定了「这台机器上的实例能不能继续跑、能用哪些模块」（通过 heartbeat 查 license_keys）。

## 四、数据关系小结

| 表/组件 | 作用 | 与授权码的衔接 |
|---------|------|----------------|
| **license_keys** | 授权码主表（管理后台生成） | 中心实体；portal 激活时写 assigned_to/activated_at；admin 作废/延期改 is_revoked/expires_at；proxy 与 heartbeat 都查此表 |
| **registry_credentials** | 门户用户 ↔ Registry 账号 | 一条记录对应一个用户+一个授权码（license_id）；激活时创建，凭证写入 htpasswd；proxy 用 registry_username 反查 license_id |
| **htpasswd** | Registry 认证 | 仅存「用户名+密码」；与 registry_credentials.registry_username 一一对应；作废授权时从 htpasswd 删除对应用户名 |
| **machine_bindings** | 授权码 ↔ 机器 | 心跳时创建/更新；同一授权码可绑多台机器（受 machine_limit 限制） |
| **instance_heartbeats** | 授权码 ↔ 运行实例 | 心跳时创建/更新；用于并发数控制与在线统计 |

**一句话**：  
- **管理后台**负责「授权码从无到有」；  
- **门户**负责「授权码从码到人、到 Registry 凭证」；  
- **镜像仓库 + proxy**负责「只认凭证、按凭证背后的授权码做拉取控制」；  
- **软件系统**用同一授权码做「拉取时的凭证」和「运行时的心跳」，从而在拉取与运行两端都与同一套授权衔接。

## 五、端到端流程（客户视角）

1. **管理员**在管理后台生成授权码，将码发给客户（或通过渠道发放）。
2. **客户**在门户注册并登录，在用户中心输入授权码并激活 → 获得 Registry 地址、用户名、密码。
3. **客户**在服务器上配置 Docker `insecure-registries`，执行 `docker login` + `docker pull`（用上述凭证）→ 由 registry-proxy 按该授权码的套餐与状态放行。
4. **客户**启动 linscio-ai 时，配置中填写**同一授权码**；应用定期调心跳接口 → 后端根据该授权码校验作废/过期/机器数/并发数，并返回模块与配额信息。
5. 若**管理员**在后台对该授权码作废或延期，则：  
   - 拉取：该授权码对应的 Registry 用户名会从 htpasswd 删除（或 proxy 查库得到 is_revoked）→ 无法再 pull；  
   - 运行：下次心跳返回 403 或 license_expired → 应用可拒绝继续运行或提示续费。

以上即门户、管理后台、镜像仓库与软件系统在授权管理上的完整衔接方式。
